# Требования к системе

Функциональный набор и ограничения. Архитектура и структура кода — в [ARCHITECTURE.md](ARCHITECTURE.md).

**Окружение:** Node.js 18+ (см. `engines` в package.json, [DEPLOY.md](../DEPLOY.md)).

## 1. Назначение

Бот для мониторинга и автоматического переноса записей на собеседование по визе США (AIS) и задел под VFS Global. Режимы: один пользователь (CLI `bot`) и мультипользовательский пул из Google Sheets (`monitor`).

## 2. Обязательные фичи

- Мультипользовательский мониторинг из хранилища (сейчас — Google Sheets).
- Ротация пользователей с приоритетом (cooldown, lastChecked, lastBooked).
- Общий кэш дат по провайдерам с TTL и персистентностью (лист "Available Dates Cache").
- Валидация дат: раньше текущей записи, в диапазонах, не раньше (today + reaction_time).
- Букинг: слоты → выбор времени → форма; уведомления об успехе/ошибке.
- Уведомления в Telegram (успех, слот, ошибка, старт монитора, квоты Sheets).
- Поддержка нескольких провайдеров (AIS, VFS): при `user.provider === 'vfsglobal'` фабрика возвращает VfsGlobalProviderAdapter; иначе — AIS. Детали запуска — в [ARCHITECTURE.md](ARCHITECTURE.md).
- Single-user режим (legacy): один пользователь из .env, целевая/минимальная дата.

## 3. Настройки

- Интервалы: refresh, sheets refresh, cache TTL, rotation cooldown.
- AIS: request delay, rate limit backoff.
- Telegram: bot token, manager chat id.
- Опционально: 2Captcha API key, facility_id. Источник: .env + переопределение из Settings.

## 4. Утилиты и команды

- `get-chat-id` — получение Telegram chat ID.
- `test-sheets` — проверка доступа к таблице.
- `test-vfs-captcha` — отладка капчи/Cloudflare для VFS.

## 5. Надёжность

- Учёт квот Google Sheets (429, retry, уведомление).
- Обработка сетевых ошибок и rate limit (socket hang up, backoff).
- Секреты только в env/хранилище.

## 6. Требования при изменениях кода

- Сохранять обязательные фичи (разделы 2–5 выше).
- Соблюдать лучшие практики (раздел 8 ниже).
- Не размывать границы слоёв (домен, приложение, порты, адаптеры — см. [ARCHITECTURE.md](ARCHITECTURE.md)).
- Держать один контур провайдеров через VisaProvider и фабрику.

## 7. Статус реализации

Обязательные фичи (разделы 2–5) реализованы. Текущее состояние структуры и слоёв — в [ARCHITECTURE.md](ARCHITECTURE.md), планы доработок — в [ROADMAP.md](ROADMAP.md).

## 8. Лучшие практики (архитектура и качество)

Зафиксированные практики обязательны при доработке кода. Детали — в [ARCHITECTURE.md](ARCHITECTURE.md) и [CODE_QUALITY.md](CODE_QUALITY.md).

### 8.1 Архитектура

- **Порты и адаптеры (hexagonal):** сценарии (use cases) зависят только от интерфейсов в `ports/`; конкретные реализации (Sheets, Telegram, AIS/VFS) — в `adapters/`. Замена источника данных или канала уведомлений не требует переписывания сценариев.
- **Composition root:** сборка зависимостей (создание адаптеров, подключение к портам) выполняется только в точке входа (например, `createMonitorContext` для monitor). Use cases и домен не создают адаптеры и не импортируют инфраструктуру напрямую.
- **Домен без I/O:** доменные сущности и правила не выполняют сетевых запросов, обращений к БД или файловой системе. Вся работа с внешним миром — через порты и адаптеры.
- **Единая точка входа для внешних сервисов:** доступ к провайдерам (AIS, VFS) — только через порт VisaProvider и фабрику (createVisaProvider). Прямое создание низкоуровневых клиентов в командах или use cases не допускается.
- **Явные зависимости:** зависимости передаются в конструктор или параметрами (dependency injection). Глобальное изменяемое состояние в lib минимизировано; где нужно — фабрики и единый экземпляр из composition root.
- **Один модуль — одна ответственность:** не смешивать в одном модуле/классе разнородные зоны ответственности (например, низкоуровневый API и доменная логика).

### 8.2 Качество кода

- **Типизация и линтинг:** код проходит `npm run typecheck` и `npm run lint`. В новых правках не добавлять `any` без обоснования. См. [CODE_QUALITY.md](CODE_QUALITY.md) § 3.
- **Обработка ошибок:** в `catch` не глотать ошибки — либо проброс (rethrow), либо логирование и осмысленная реакция (retry, уведомление, выход).
- **Логирование:** структурированный логгер (pino), уровни `debug`/`info`/`warn`/`error`. Секреты не логировать; сырые stack в продакшене — только в обёртке.
- **Секреты:** пароли, токены, API-ключи — только в env или хранилище, не в коде и не в репозитории.
- **Контракты и ADR:** публичные порты описаны в [CONTRACTS.md](CONTRACTS.md); при изменении портов — обновить контракты и тесты с моками. Критичные архитектурные решения — в [ADR](adr/README.md).
- **Перед merge:** выполнять `npm run lint`, `npm run typecheck`, `npm test`; при изменении портов — обновить документацию и тесты.
