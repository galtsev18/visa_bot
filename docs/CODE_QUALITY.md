# Требования к качеству кода

Стандарты кодирования, линтинг, типизация и ревью. Связано: [REQUIREMENTS.md](REQUIREMENTS.md) · [TESTING.md](TESTING.md) · [ARCHITECTURE.md](ARCHITECTURE.md).

---

## 1. Общие принципы

- Соблюдать границы слоёв: домен без I/O; use cases зависят только от портов; адаптеры реализуют порты (см. [REQUIREMENTS.md](REQUIREMENTS.md) § 6, [ARCHITECTURE.md](ARCHITECTURE.md)).
- Избегать размывания ответственности: один модуль/класс — одна зона ответственности.
- Предпочитать явные зависимости и параметры глобальному состоянию (см. composition root).

---

## 2. Стиль и форматирование

- **Prettier:** единый формат (см. `.prettierrc`). Перед коммитом рекомендуется запускать `npm run format` или включить format on save.
- **Стиль:** одинарные кавычки, точка с запятой, trailing comma (es5), printWidth 100, tabWidth 2.
- Имена: camelCase для переменных и функций; PascalCase для классов и типов; UPPER_SNAKE для констант конфигурации.

---

## 3. Линтинг и типизация

- **ESLint:** конфигурация в `eslint.config.js`. Правила TypeScript (typescript-eslint) применяются к `.ts` файлам. Игнорируются `dist/`, `node_modules/`.
- **TypeScript:** строгая проверка для кода в `src/ports/`, `src/application/`, `src/adapters/`, `src/composition/`. Команда: `npm run typecheck`. В `lib/` и `commands/` допустим JS с JSDoc при необходимости.
- Требования к линту и typecheck: `npm run lint` и `npm run typecheck` проходят в CI без ошибок. Предупреждения (warn) допустимы, но их количество не должно расти без обоснования.

---

## 4. Обработка ошибок и логирование

- Не глотать ошибки: в catch либо проброс (rethrow), либо логирование и осмысленная реакция (уведомление, retry, выход).
- Логирование: использовать структурированный логгер (pino). Уровни: `debug`, `info`, `warn`, `error`. В продакшене не выводить в консоль сырые stack без обёртки.
- Секреты (пароли, токены, API keys) не логировать и не попадают в код — только env/хранилище (см. REQUIREMENTS § 5).

---

## 5. Зависимости

- Новые зависимости обосновывать. Предпочитать малый набор проверенных пакетов.
- Production-зависимости — только необходимое для работы; dev-зависимости — линтеры, тесты, сборка. В `package.json` фиксировать версии в lockfile; при обновлении проверять тесты и линт.

---

## 6. Документация кода

- Публичные порты (интерфейсы в `src/ports/`): описание назначения и контракта в JSDoc/комментариях; полная спецификация — в [CONTRACTS.md](CONTRACTS.md).
- Сложная бизнес-логика (валидация дат, ротация): краткий комментарий или ссылка на REQUIREMENTS/ADR.
- Не дублировать очевидное; комментарии должны объяснять «зачем», а не «что».

---

## 7. Ревью и CI

- Перед merge: код должен проходить `npm run lint`, `npm run typecheck`, `npm test`.
- При изменении портов или контрактов: обновить CONTRACTS.md и при необходимости ADR; тесты с моками портов — обновить.
- Критичные архитектурные решения фиксировать в [ADR](adr/README.md).
