# ADR-0000: Use ADR for architecture decisions

**Status:** Accepted  
**Date:** 2025-03  
**Deciders:** Команда проекта

## Context

Нужна документация по архитектуре «по-взрослому»: не только описание текущего состояния, но и фиксация ключевых решений и их обоснований, чтобы новые участники и будущие изменения опирались на единый формат.

## Decision

Используем **Architecture Decision Records (ADR)** в папке `docs/adr/` в формате Markdown. Каждый ADR — один файл с номером и коротким названием (`XXXX-short-title.md`). В каждом ADR обязательны: Status, Date, Context, Decision, Consequences.

## Consequences

### Положительные

- История решений в одном месте, легко искать по номерам и названиям.
- Контекст и последствия явно зафиксированы.
- Можно помечать устаревшие решения (Deprecated / Superseded).

### Отрицательные / риски

- Нужно не забывать добавлять ADR при значимых решениях.

### Нейтральные

- Оглавление в `docs/adr/README.md` с таблицей ADR.
