# Требования к тестированию

Политика тестов, типы тестов и критерии приёмки. Связано: [REQUIREMENTS.md](REQUIREMENTS.md) · [CONTRACTS.md](CONTRACTS.md) · [CODE_QUALITY.md](CODE_QUALITY.md).

---

## 1. Цели тестирования

- Обеспечить регрессию при рефакторинге (домен, порты, use cases).
- Проверить контракты и граничные условия без зависимости от внешних сервисов (где возможно).
- Документировать ожидаемое поведение через тесты.

---

## 2. Уровни тестов

| Уровень | Область | Инструмент | Внешние зависимости | Обязательность |
|---------|---------|------------|---------------------|----------------|
| **Unit** | Домен (User, userRotation), валидация дат, парсеры | Node test runner + tsx | Нет | Обязательно для новой/меняемой доменной логики |
| **Unit** | Use cases (startMonitor, checkUserWithCache, attemptBooking) | Node test runner + tsx | Моки портов | Желательно при изменении сценариев |
| **Unit** | Адаптеры (обёртки над портами) | Node test runner + tsx | Моки внешних API или тестовые данные | По возможности |
| **Integration** | Репозиторий + Sheets, провайдер + AIS/VFS, Telegram | Node test runner | Реальные или тестовые сервисы | Опционально (см. ROADMAP) |
| **E2E** | Полный цикл monitor/bot | Ручной / скрипт | Реальные конфиг и сервисы | Не автоматизировано |

---

## 3. Требования к unit-тестам

- **Домен:** тесты на `User.isDateValid`, `isDateEarlierThanCurrent`, `isDateInRange`, `isDateAfterReactionTime`, `needsAppointment`; на `getNextUser` (ротация по приоритету, cooldown). Без I/O.
- **Use cases:** вызов сценария с моками (UserRepository, DateCache, NotificationSender, VisaProvider/Bot). Проверка: вызовы моков в ожидаемом порядке и с ожидаемыми аргументами; возвращаемые значения.
- **Именование:** `test/` или `*.test.ts` / `*.test.js` рядом с модулем или в каталоге `test/`. Описательные имена: `isDateValid returns false when date is after current`.
- **Изоляция:** один тест не должен зависеть от другого; моки сбрасывать или создавать заново в каждом тесте при необходимости.

---

## 4. Моки и контракты

- Моки портов должны соответствовать интерфейсам из `src/ports/` (см. [CONTRACTS.md](CONTRACTS.md)). При изменении порта все тесты, использующие мок этого порта, должны быть обновлены.
- Рекомендуется использовать один и тот же объект-мок для нескольких вызовов в сценарии (например, `repo.getActiveUsers` возвращает заданный список пользователей), чтобы тест оставался читаемым.

---

## 5. Запуск и CI

- **Локально:** `npm test` — запуск всех тестов (Node test runner с поддержкой .ts через tsx).
- **CI:** тесты выполняются в pipeline (см. `.github/workflows/ci.yml`). Падение тестов блокирует прохождение.
- Код, попадающий в main/master, должен проходить `npm test` (и при необходимости добавлять тесты при изменении поведения).

---

## 6. Покрытие

- Требование к минимальному покрытию не задано. При добавлении или изменении доменной логики и use cases ожидается добавление или обновление соответствующих тестов.
- Опционально: подключить инструмент измерения покрытия (c8, nyc) и отслеживать тренд в CI.

---

## 7. Интеграционные тесты

- Текущая политика: не обязательны, в [ROADMAP.md](ROADMAP.md) указаны как возможное улучшение.
- При введении интеграционных тестов: использовать тестовую таблицу Google Sheets и/или тестовый Telegram-бот / заглушки; не хранить секреты в репозитории; явно помечать такие тесты (например, `test:integration` или отдельная папка `test/integration/`).
